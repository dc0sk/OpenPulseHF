# OpenPulseHF Review — 2026-05-08

## Executive summary
- The wire-format spec and decoder disagree on compatibility policy: the docs say unknown message types should be ignored and higher versions rejected, but the decoder rejects unknown message types and silently accepts unknown versions.
- `CLAUDE.md` is stale: the active phase annotation is outdated, and the crate map omits a large set of workspace crates that are already functional.
- The QSY trust allowlist is dead configuration today, and the QSY session state machine has no graceful path when the initiator and responder candidate lists have no overlap.
- Several high-risk claims checked out in code and test runs: rate adaptation, relay forwarding, wire codecs, QSY frames/scanner, and the Phase 9 UI/runtime pieces are present and exercised by tests.

## Fabrication Risk
My read is that the review is mostly grounded, but there are a few places where the confidence level is lower than the wording implies.

Strongly supported claims:
- The targeted test runs succeeded for rate adaptation, QSY, relay forwarding, wire codecs, and query propagation.
- The QSY frame/state-machine/scanner code is real and not a stub.
- The relay, mesh, daemon, repeater, DSP, panel, and testmatrix crates all have concrete runtime code.

Weaker or inference-heavy claims:

## Comments (DC0SK / Claude, 2026-05-08)

### Overall assessment

The review is thorough and the findings are well-grounded. The fabrication-risk section
is a useful addition — it correctly identifies where the reviewer is making inferences vs.
reading code. No finding contradicts what we know about the codebase.

### Agreement

All four recommended actions are valid. Items 1 and 4 are pure documentation hygiene and
have no risk. Items 2 and 3 both require a code decision before writing anything.

### On finding 2 — wire format compatibility (nuance)

The review says code and spec disagree and recommends deciding which is right.
**The code is the right one.** The comment at `wire_query.rs:115` makes the intent
explicit: versions are forward-compatible by design. Returning an error on unknown message
types is also the correct posture for a binary protocol — silently ignoring unknown types
causes confused state machines and missed frames, which is exactly what the `UnknownMsgType`
error prevents. The spec was written aspirationally; the code reflects the implemented
reality. The right action is to update the spec, not loosen the decoder.

The 0x09/0x0A extension (`RigCtrlCmd`, `BroadcastFrame`) is a straightforward spec gap:
the code shipped these types after the spec was last edited. Documenting them in the spec
table is the only fix needed.

### On finding 3 — QSY disjoint-candidate path (more serious than noted)

The review flags this as a brittleness concern with no failing test. Digging deeper: when
`pick_best_freq` returns `Err(QsyError::InvalidTransition(...))` and `apply()` propagates
it, the session state is left in `Listed` — it did **not** transition to `Rejected` and it
did **not** send `QSY_REJECT` to the peer. The peer will wait forever for a `QSY_ACK` that
never arrives. This is a protocol hang, not just a recoverable error. It warrants fixing as
a correctness issue: when the intersection is empty, `apply()` should send
`[SendFrame(QSY_REJECT), Reject{...}]` and move to `Rejected`, the same path as
`policy.enabled == false`.

### On finding 3 — QSY trust allowlist

We already annotated the config field as "reserved / not yet enforced" in the post-review
commit. The comment about the roadmap mentioning it as a gate is valid but this is clearly
a planned future feature, not a shipped regression. Treating it as deferred work is
appropriate.

### On finding 6 — CLAUDE.md active phase

The stale "Phase 2" annotation has been wrong for many phases. The correct current label
is effectively "Phase 9 / FF series" but the CLAUDE.md model of naming a single active
phase breaks down now that multiple phase tracks are running in parallel. The fix should
restructure the section into "completed" and "active tracks" rather than just updating the
number.

### On Phase 9 roadmap markers

The review correctly identifies Phase 9 items as implemented but unmarked ✅. This is a
straightforward documentation update — the items were built as part of prior PRs but the
roadmap was never updated to close them out.

## Actions

Ranked by priority.

### P1 — Fix QSY disjoint-candidate protocol hang *(correctness, one session — small)*

**Problem**: `apply(QSY_VOTE)` propagates `pick_best_freq` error as `Err`, leaving the
session in `Listed` state without sending `QSY_REJECT`. Peer hangs waiting for `QSY_ACK`.

**Fix**: In the `QsyFrame::Vote` arm of `apply()`, catch `Err` from `pick_best_freq` and
convert it to `[SendFrame(QSY_REJECT), Reject{reason}]` + `State::Rejected`, matching the
policy-rejection pattern already used in the `QSY_REQ` handler. Add a dedicated test:
`disjoint_candidate_lists_emits_reject`.

**Files**: `crates/openpulse-qsy/src/session.rs`, `crates/openpulse-qsy/tests/qsy_session.rs`
