## Executive summary

Both open items (F-1, F-2) from review-260524 were resolved in PR #294. No regressions.

PR #295 introduced the QSY enable toggle and bandplan selector into the panel and TUI.
Two minor findings remain:

- `QsyConfig::default()` initialises `bandplan_mode` to the deprecated `"ham-iaru"` string,
  which is not in the `BANDPLAN_OPTIONS` table used by the panel dropdown. A fresh install
  with no config file displays "Unrestricted" as the current bandplan (via the `unwrap_or`
  fallback) even though the default policy is `HamIaruRegion1`, not unrestricted.
- `SetConfig` commands received by the daemon handler update in-memory shared state but do
  not call `save_qsy_config()`. Changes made through the panel survive for the current
  daemon session only; a daemon restart restores config from the TOML file. The TUI
  saves directly and is unaffected. This asymmetry is silent — no confirmation or warning
  is shown to the operator.

No blocking issues. Both are minor UX/correctness problems addressable in a single small PR.

## New findings

### F-1 (minor): `QsyConfig::default()` sets deprecated `"ham-iaru"` bandplan mode

**Finding**: `QsyConfig::default()` at
[lib.rs:386](../../../crates/openpulse-config/src/lib.rs#L386) sets
`bandplan_mode: "ham-iaru".into()`.  The string `"ham-iaru"` corresponds to the
deprecated `BandplanMode::HamIaru` variant, which was deprecated in the PR #258 batch.
It is not included in `BANDPLAN_OPTIONS` in the panel
([app.rs:40–45](../../../apps/openpulse-panel/src/app.rs#L40)):

```rust
const BANDPLAN_OPTIONS: &[(&str, &str)] = &[
    ("unrestricted", "Unrestricted"),
    ("ham-iaru-r1", "IARU Region 1"),
    ("ham-iaru-r2", "IARU Region 2"),
    ("ham-iaru-r3", "IARU Region 3"),
];
```

`bandplan_label("ham-iaru")` falls through to `unwrap_or("Unrestricted")`, so the panel
shows "Unrestricted" as the active bandplan for a fresh installation that has
`bandplan_awareness_enabled: true` and `bandplan_mode: "ham-iaru"`. The operator
may believe bandplan guardrails are off when they are in fact active (using the
deprecated `HamIaru` dispatch path).

**Recommended fix**: Change `QsyConfig::default()` to `bandplan_mode: "ham-iaru-r1".into()`
to match `BandplanPolicy::default()` which already uses `HamIaruRegion1`.  Single-line
change; no test changes required (the default is not asserted in existing tests).

**File**: `crates/openpulse-config/src/lib.rs`

---

### F-2 (minor): `SetConfig` from the panel is ephemeral — not persisted to disk

**Finding**: The daemon's `handle_command` helper at
[lib.rs:534–543](../../../crates/openpulse-daemon/src/lib.rs#L534) updates the four
in-memory `Arc<Mutex<_>>` guards when it receives `SetConfig`, but does not call
`openpulse_config::save_qsy_config()` or otherwise write to the TOML file.

The panel sends `SetConfig` when the operator clicks "Apply" in the Config window
([app.rs:654–656](../../../apps/openpulse-panel/src/app.rs#L654)):

```rust
if ui.button("Apply").clicked() {
    self.send(ControlCommand::SetConfig {
        config: self.config_draft.clone(),
```

The TUI calls `openpulse_config::save_qsy_config()` directly from the TUI process
([main.rs](../../../crates/openpulse-tui/src/main.rs)) on the `Q` and `b` keystrokes, so
TUI changes survive a restart. Panel changes do not.

**Impact**: An operator who uses the panel to enable QSY or change the bandplan, then
restarts the daemon, will find those changes reverted silently.

**Recommended fix**: In `handle_command`, after updating the in-memory shared state,
call `openpulse_config::save_qsy_config(*qsy, &*bp)` for `SetConfig` commands that
change `qsy_enabled` or `bandplan_mode`. `save_qsy_config` is already designed for
this use case. The call must happen after the locks are released (or pass the values by
copy before dropping the guards) to avoid holding the async lock across a sync I/O call.

**Files**: `crates/openpulse-daemon/src/lib.rs`
