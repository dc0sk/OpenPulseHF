All nine recommended actions from review-260517 are resolved. The fixes are
correct and well-tested. Two minor follow-up items remain: the preamble length-mismatch
log is at `debug` level (invisible at default `info`) for a path that silently disables

## New findings

### F-1 (minor): Preamble mismatch logged at `debug`, not `warn`

**Finding**: `correlate_bpsk` at
[preamble.rs:217](../../../crates/openpulse-dsp/src/preamble.rs#L217) emits a
`tracing::debug!` event on length mismatch, then returns `(0.0, 0.0)`. At the default
`info` log level — and at the `error`-only level used in the benchmark harness — this
event is invisible. A caller that passes a wrong-length receive buffer will silently
receive zero correlation, meaning preamble detection is silently disabled with no
observable signal outside of `RUST_LOG=debug` runs.

**Recommended fix**: Promote to `tracing::warn!`. Preamble detection being skipped is
an abnormal condition that warrants operator attention regardless of log level. The
message already includes `expected_len` and `received_len` which are sufficient for
diagnosis.

### F-2 (minor): `DopplerTracker::get_doppler_rate()` sign convention undocumented

**Finding**: `DopplerTracker::get_doppler_rate()` at
[doppler_tracker.rs:116](../../../crates/openpulse-dsp/src/doppler_tracker.rs#L116)
returns a signed `f32`. The P7 fix added sign-convention documentation to
`AdaptiveAfcLoopBandwidth` (the consumer), but not to the producer. A future caller
reading only the getter's signature and doc comment cannot determine whether positive
values represent an upward or downward Doppler shift.

**Recommended fix**: Add a one-line doc to `get_doppler_rate`: `/// Returns the
signed Doppler rate estimate in rad/sample (positive = increasing carrier phase,
i.e. upward Doppler shift).`
