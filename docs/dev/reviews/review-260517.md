**Review window**: PRs #216–#281 (2026-05-13 to 2026-05-17)

## Executive summary

- `occupied_bandwidth_hz` in the bandplan module is missing every active `-RRC` mode variant
  and `SCFDMA52-64QAM-P4`, causing those modes to return `UnknownOperatingMode` instead of
  a passing check when bandplan guardrails are active.
- `should_hold_ack_up_without_snr_candidate` in the engine identifies the wideband-HD profile
  by hard-coding specific mode strings at Sl12–Sl14; a profile rename silently disables the
  SNR gate without a compile error.
- Four new DSP history buffers use `Vec::remove(0)` (O(n) eviction) where `VecDeque` is the
  correct type.
- `BandplanPolicy::default()` sets `mode` to the deprecated `HamIaru` variant — the same
  deprecation added in this batch of PRs.
- The roadmap's profile scope table lists SL7 as `SCFDMA52-QPSK` but the actual `hpx_hf()`
  profile maps SL7 to `8PSK500`; the mode string `SCFDMA52-QPSK` does not exist in the plugin.

## Findings by topic

### 1. Bandplan guardrails (`crates/openpulse-qsy/src/bandplan.rs`)

#### 1a. Missing `-RRC` mode variants in `occupied_bandwidth_hz`

**Finding (Bug)**: `occupied_bandwidth_hz` at [bandplan.rs:248](../../../crates/openpulse-qsy/src/bandplan.rs#L248)
is the single lookup that converts a mode string to an occupied bandwidth estimate for
policy checks. It is missing every `-RRC` suffix variant that is actively registered and
tested in the plugin layer:

| Missing mode | Plugin | BW estimate needed |
|---|---|---|
| `BPSK250-RRC` | bpsk-plugin ([plugins/bpsk/src/lib.rs:85](../../../plugins/bpsk/src/lib.rs#L85)) | ≈ 350 Hz |
| `QPSK500-RRC` | qpsk-plugin | ≈ 675 Hz |
| `QPSK1000-HF-RRC` | qpsk-plugin | ≈ 1350 Hz |
| `8PSK500-RRC` | psk8-plugin ([plugins/psk8/src/lib.rs:34](../../../plugins/psk8/src/lib.rs#L34)) | ≈ 675 Hz |
| `8PSK1000-RRC` | psk8-plugin ([plugins/psk8/src/lib.rs:35](../../../plugins/psk8/src/lib.rs#L35)) | ≈ 1350 Hz |
| `8PSK1000-HF-RRC` | psk8-plugin ([plugins/psk8/src/lib.rs:33](../../../plugins/psk8/src/lib.rs#L33)) | ≈ 1350 Hz |
| `8PSK2000-RRC` | psk8-plugin ([plugins/psk8/src/lib.rs:38](../../../plugins/psk8/src/lib.rs#L38)) | ≈ 2700 Hz |
| `SCFDMA52-64QAM-P4` | scfdma-plugin | ≈ 3200 Hz |

`occupied_bandwidth_hz` currently has `QPSK9600-RRC`, `8PSK9600-RRC`, and `64QAM2000-RRC`
in the table but not any of the eight modes above. When `BandplanPolicy::validate_frequency`
calls `occupied_bandwidth_hz` for one of these modes, it returns `None`, which is mapped to
`BandplanError::UnknownOperatingMode` at [bandplan.rs:158](../../../crates/openpulse-qsy/src/bandplan.rs#L158)
and [bandplan.rs:216](../../../crates/openpulse-qsy/src/bandplan.rs#L216). The transmission is
rejected even though the modes are HF-legal by design. This is fail-safe in the wrong
direction: legitimate HF-RRC transmissions are blocked while the modes they were designed to
replace (wider non-RRC variants) pass.

**Recommended fix**: Add the missing modes to the match block. The RRC bandwidth is
`(1 + 0.35) × baud_rate × 1000/8000` Hz (α = 0.35, 8 kHz sample rate). Concrete additions:

```rust
"BPSK250-RRC"                  => Some(350),
"QPSK500-RRC"                  => Some(675),
"QPSK1000-HF" | "QPSK1000-HF-RRC" => Some(1350),
"8PSK500-RRC"                  => Some(675),
"8PSK1000-RRC" | "8PSK1000-HF-RRC" => Some(1350),
"8PSK2000-RRC"                 => Some(2700),
"SCFDMA52-64QAM-P4"            => Some(3200),
```

#### 1b. `BandplanPolicy::default()` uses the deprecated `HamIaru` variant

**Finding (Minor)**: [bandplan.rs:58–67](../../../crates/openpulse-qsy/src/bandplan.rs#L58) — the
`Default` impl sets `mode: BandplanMode::HamIaru` under an `#[allow(deprecated)]` suppressor.
This was added precisely to force callers toward region-specific variants, yet the canonical
default remains the deprecated one. Any code constructing `BandplanPolicy::default()` and then
inspecting or logging `.mode` will surface the deprecated variant.

**Recommended fix**: Change the default to `BandplanMode::HamIaruRegion1` (the most conservative
choice from a spectrum-width standpoint), or require callers to supply the mode explicitly.

#### 1c. Region 3 silently maps to Region 1

**Finding (Known limitation, documentation gap)**: `find_region3_band` and `find_region3_segment`
at [bandplan.rs:441–447](../../../crates/openpulse-qsy/src/bandplan.rs#L441) unconditionally
delegate to Region 1 functions, with a comment acknowledging Region 3 varies by country.
Operators in Japan, Australia, New Zealand, or China face different digital segment allocations
that `HamIaruRegion3` does not enforce. The limitation is not surfaced to the user.

**Recommended fix**: Document explicitly in the CLI help and in `BandplanError::FrequencyOutOfBand`
that `HamIaruRegion3` uses Region 1 allocations as a conservative proxy. Consider adding a
`BandplanError::ApproximateRegion` variant or a warning-level log at validate time.

### 2. Engine: wideband-HD SNR gate (`crates/openpulse-modem/src/engine.rs`)

**Finding (Fragile)**: `should_hold_ack_up_without_snr_candidate` at
[engine.rs:342–364](../../../crates/openpulse-modem/src/engine.rs#L342) identifies the
wideband-HD profile by comparing mode strings at Sl12, Sl13, and Sl14:

```rust
let is_wideband_hd = profile.mode_for(SpeedLevel::Sl12) == Some("SCFDMA52-64QAM-P4")
    && profile.mode_for(SpeedLevel::Sl13) == Some("SCFDMA52-64QAM")
    && profile.mode_for(SpeedLevel::Sl14) == Some("64QAM2000-RRC");   // engine.rs:356–359
```

If `hpx_wideband_hd()` in [profile.rs:301–302](../../../crates/openpulse-core/src/profile.rs#L301)
is ever updated (mode names renamed, slots shifted, ceiling changed), this check silently
becomes a no-op or a false positive at runtime with no compile-time signal. The SNR admission
gate for SL14 could either be permanently bypassed or applied to the wrong profile.

**Recommended fix**: `SessionProfile` should expose an explicit predicate or a field, e.g.:

```rust
pub snr_gated_levels: Vec<SpeedLevel>,   // in SessionProfile
```

populated by `hpx_wideband_hd()` and consulted by the engine without inspecting mode strings.
The engine then becomes `profile.is_snr_gated(tx_level)` — correct under any future profile
change.

### 3. New DSP modules — `Vec::remove(0)` instead of `VecDeque`

**Finding (Style/efficiency)**: Four ring-buffer histories in the new DSP modules use
`Vec<f32>` with `Vec::remove(0)` to evict the oldest entry. This is O(n) per update because
it shifts the entire buffer on every call. The correct type is `VecDeque<f32>` with
`pop_front`/`push_back`.

| Location | Field | Line |
|---|---|---|
| [preamble.rs](../../../crates/openpulse-dsp/src/preamble.rs#L231) | `correlation_history` | 231 |
| [preamble.rs](../../../crates/openpulse-dsp/src/preamble.rs#L250) | `phase_history` | 250 |
| [doppler_tracker.rs](../../../crates/openpulse-dsp/src/doppler_tracker.rs#L46) | `phase_history` | 46 |
| [snr_hysteresis.rs](../../../crates/openpulse-core/src/snr_hysteresis.rs#L129) | `snr_history` | 129 |

For the small sizes in use (≤ 32 entries), this is not a performance problem today, but
`VecDeque` is the semantically correct type and should be used to avoid a footgun if history
length grows.

### 4. `crates/openpulse-dsp/src/preamble.rs` — silent length mismatch

**Finding (Debuggability)**: `correlate_bpsk` at
[preamble.rs:214–216](../../../crates/openpulse-dsp/src/preamble.rs#L214) returns `(0.0, 0.0)`
when `received.len() != self.reference.len()`. Zero correlation is indistinguishable from a
genuine null-correlation result. If frame boundary drift or misconfiguration causes a size
mismatch, callers will silently see no lock, with no diagnostic path.

**Recommended fix**: Add a `tracing::debug!` or `tracing::warn!` log at the mismatch branch,
or change the return type to `Option<(f32, f32)>` so callers can distinguish "no correlation
computed" from "correlation was zero".

### 5. `crates/openpulse-dsp/src/doppler_tracker.rs` — signed vs. absolute Doppler rate

**Finding (Minor)**: `AdaptiveAfcLoopBandwidth::update` at
[doppler_tracker.rs:167](../../../crates/openpulse-dsp/src/doppler_tracker.rs#L167) stores
`doppler_rate.abs()`, discarding the sign. The private field `doppler_rate` and the public
getter `get_doppler_rate()` at
[doppler_tracker.rs:194](../../../crates/openpulse-dsp/src/doppler_tracker.rs#L194) return a
magnitude, but the name implies a signed rate. A caller expecting a signed Doppler rate (e.g.,
to determine whether drift is positive or negative) would be silently wrong.

**Recommended fix**: Rename to `doppler_rate_magnitude` or document the return value as
absolute in the getter doc comment.

### 6. `crates/openpulse-core/src/tx_metadata.rs` — `TxSessionLog` accepts cross-station frames

**Finding (Regulatory logging risk)**: `TxSessionLog::new(station_id)` sets a top-level
`station_id` at [tx_metadata.rs:88–92](../../../crates/openpulse-core/src/tx_metadata.rs#L88),
but `log_frame` at [tx_metadata.rs:96](../../../crates/openpulse-core/src/tx_metadata.rs#L96)
appends any `TxMetadata` without checking that its `station_id` matches the log's. A caller
could create `TxSessionLog::new("W5ABC")` and then log frames from "K7XYZ" with no error.
For a regulatory-compliance log this is a risk: the log-level callsign and the per-frame
callsign can silently diverge.

**Recommended fix**: Either enforce in `log_frame` that `metadata.station_id == self.station_id`,
returning a `Result`, or remove the top-level `station_id` field and derive it from the frames
when serialising (using the first frame's callsign, asserting all are equal).

### 7. `crates/openpulse-cli/src/commands/session_metrics.rs` — throughput proxy

**Finding (Minor, documented)**: `BYTES_PER_SUCCESS_FRAME_PROXY = 223.0` at
[session_metrics.rs:9](../../../crates/openpulse-cli/src/commands/session_metrics.rs#L9) is the
RS codec capacity, not a measured payload size. The computed `throughput_bps` at
[session_metrics.rs:99](../../../crates/openpulse-cli/src/commands/session_metrics.rs#L99)
will be inflated for sessions carrying short payloads (ACK/beacon frames, short messages,
test payloads). The `notes` array in the JSON output correctly documents this, but the field
name `throughput_bps` alone is misleading — it will be read as a measured value by
downstream consumers.

**Recommended fix**: Rename to `throughput_bps_upper_bound` or add a sibling field
`throughput_bps_note: "223-byte RS capacity proxy; actual payload may be smaller"` adjacent
to the value rather than only in a notes array that may be filtered.

### 8. `crates/openpulse-cli/src/commands/mode_advisor.rs` — hardcoded SL level list

**Finding (Minor)**: `recommend_hf_level` at
[mode_advisor.rs:33–40](../../../crates/openpulse-cli/src/commands/mode_advisor.rs#L33) iterates
a fixed array `[Sl2, Sl3, Sl4, Sl5, Sl6, Sl7]`. This list mirrors the current `hpx_hf()`
ceiling of SL7. If the profile ceiling is raised or the profile is extended, this list needs
a manual update; there is no compile-time enforcement that it tracks the profile.

**Recommended fix**: Replace the hardcoded array with an iterator over `SpeedLevel::all()`
(or equivalent), stopping at the first level for which `profile.mode_for(level)` returns
`None`. This automatically follows profile changes.

### 9. `plugins/scfdma/tests/pilot_density_review.rs` — sequential seed spacing

**Finding (Minor)**: `WattersonConfig::poor_f1(Some(0x7200 + frame as u64))` at
[pilot_density_review.rs:48](../../../plugins/scfdma/tests/pilot_density_review.rs#L48) uses
linearly sequential seeds. If the Watterson channel has any periodic behavior correlated at
period 1, consecutive seeds produce correlated samples, reducing the effective sample count
below the nominal 24 frames.

**Recommended fix**: Use a multiplicative seed step, e.g.,
`0x7200u64.wrapping_mul(0x9E3779B97F4A7C15).wrapping_add(frame as u64)`, to spread seeds
across the seed space more uniformly.

`SCFDMA52-64QAM` already appears) or add a separate SC-FDMA modes section.
