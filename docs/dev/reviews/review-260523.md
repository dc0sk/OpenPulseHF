## Executive summary

Following a thorough workspace audit and test-run analysis:
1. **Compilation status**: The workspace compiles successfully under `cargo check --workspace` and the test suite passes cleanly under `cargo test --workspace --no-default-features`.
2. **Clippy blocks**: Several new Clippy warnings (treated as errors in CI via `-- -D warnings`) are present across the workspace crates. These prevent clean CI pipeline validation and must be addressed.
3. **Convention violations**: Five occurrences of `.expect()` or `.unwrap()` were found in library production paths, violating the strict `CLAUDE.md` guideline designed to prevent library-level panics. One of these (`ofdm_sim.rs:375` using `.unwrap()` on `partial_cmp`) poses a high operational risk if NaN values occur in signal processing.
4. **Current branch changes**: The current branch `fix/sharp-edges-fec-watterson` relaxes several BER thresholds for slow-fading channels (Watterson F1) in tests. This is mathematically correct due to deep coherence fades, but needs to be finalized and checked in.

### Topic 2 — Panic Risks and CLAUDE.md Violations

Library crates must remain panic-free in their production paths. Several unwraps/expects were found in production files.

#### F-2.1: `partial_cmp().unwrap()` Panic Risk (High)
- **File**: [ofdm_sim.rs:375](file:///home/dc0sk/git/OpenPulseHF/crates/openpulse-modem/src/ofdm_sim.rs#L375)
- **Code**: `.max_by(|(_, a), (_, b)| a.abs().partial_cmp(&b.abs()).unwrap())`
- **Impact**: High. If any signal samples become NaN (e.g. from numerical issues in fading simulation), `partial_cmp` returns `None` and the call to `.unwrap()` will panic the entire application/modem process. Recommend using `f32::total_cmp(&a.abs(), &b.abs())` or `total_cmp` which handles NaN values safely.

#### F-2.2: `SarReassembler::ingest` expectation
- **File**: [sar.rs:164](file:///home/dc0sk/git/OpenPulseHF/crates/openpulse-core/src/sar.rs#L164)
- **Code**: `.expect("received == total guarantees all present")`
- **Impact**: Medium. Although logically correct (since `received == total` guarantees all chunks have been populated), it is still an `.expect()` in the core library path. Should handle cleanly by propagating a `SarError` or using `.ok_or()` to avoid latent panic paths.

#### F-2.3: `DecapsulationKey::to_seed` expectation
- **File**: [pq_handshake.rs:156](file:///home/dc0sk/git/OpenPulseHF/crates/openpulse-core/src/pq_handshake.rs#L156)
- **Code**: `let dk_seed = dk.to_seed().expect("freshly generated key has seed");`
- **Impact**: Medium. A library function shouldn't expect. If the underlying decapsulation key lacks an exportable seed for any reason, the program will crash.

#### F-2.4: Viterbi traceback unwrap
- **File**: [conv.rs:107](file:///home/dc0sk/git/OpenPulseHF/crates/openpulse-core/src/conv.rs#L107)
- **Code**: `let best_state = (0..STATES).min_by_key(|&s| metrics[s]).unwrap() as u32;`
- **Impact**: Low. Since `0..STATES` has a fixed size of 4, it is never empty and `unwrap()` is logically safe. Still, it violates the zero-unwrap library rule.

#### F-2.5: Trait version parsing expectations
- **File**: [plugin.rs:219, 222, 227](file:///home/dc0sk/git/OpenPulseHF/crates/openpulse-core/src/plugin.rs#L219)
- **Code**: `.expect("BUG: PLUGIN_TRAIT_VERSION must be semver")`
- **Impact**: Low. Since `PLUGIN_TRAIT_VERSION` is hardcoded in the crate, it won't change at runtime. However, it represents a minor deviation from coding standards.

### Topic 3 — Test Calibration and Watterson F1 Simulation

The branch `fix/sharp-edges-fec-watterson` relaxes K-estimator and BER thresholds across fading-channel tests:
- `llr_weighting_adaptation.rs`: Estimated Rician-K bounds expanded from `(-2.0..=8.0)` to `(-2.0..=10.0)`.
- `psk8/src/demodulate.rs`: BER threshold raised from `0.12` to `0.30` and `0.25` to `0.35` for moderate fading.
- `qpsk/src/demodulate.rs`: Fading channel degradation allowance expanded from `0.03` to `0.05`.

**Verdict**: The relaxation of these thresholds is scientifically sound. In HF fading profiles (like Watterson F1), deep slow fades trigger temporary SNR drops that drive short-term BER higher in uncoded mode. Correcting the envelope resolution in HEAD exposes these deeper slow fades correctly, requiring the tests to be recalibrated to reflect physical reality rather than artificial stability.
