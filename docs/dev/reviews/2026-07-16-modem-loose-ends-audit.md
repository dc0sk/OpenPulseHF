# OpenPulseHF — `openpulse-modem` Audit: What Isn't Nailed Down

## Executive summary

The modem core is **broadly solid**. Nothing here is a memory-safety bug, an RF-derived-input DoS, or a decode-correctness failure on the happy path — the RS-decode hardening from the prior audit round holds, and the FEC/HARQ/ARQ machinery does what it claims on the paths that are actually exercised. The real gaps cluster in three places, all confirmed against source:

1. **The OTA (receiver-led) rate ladder has a split-brain wiring problem.** The daemon runs `OtaRateController`, but the operator-facing tuning knobs (`ota_aggressiveness`, `ota_min_backlog`, `ota_upgrade_hold_frames`) and the CSMA/backlog gate logic all live in a *different*, never-activated subsystem (`RateAdaptationPolicy`). Config that looks like it's doing something is inert, and the daemon logs a false confirmation that it applied.
2. **The OTA candidate-decode loop is missing a seam guard its sibling has.** `decode_burst` explicitly sets `input_prerouted` to stop the stateful RX front-end (AGC/notch/DCD) from re-running per scan slice — a fix from a prior audit. `ota_decode_and_ack` never got the same guard, so with AGC or notch enabled, a burst gets front-end-processed 1+N times, and the notch persistence counter can trip prematurely (spurious QSY recommendation). It's opt-in-gated (default off) and there's no evidence it breaks decode itself.
3. **The synchronous OTA ACK-wait can freeze the daemon's control loop for up to ~36 s** when a peer goes silent, because `ota_send_with_ptt`'s retry loop runs directly on the `tokio::select!` arm with no cancellation check between attempts.

Everything else is lower-stakes: a few dead public methods/fields, some real-but-narrow test-coverage gaps (FSK4-ACK noise robustness, MFSK16 ACK-MAC forgery, an `#[ignore]`d acceptance test), one docs-drift item where `CLAUDE.md`'s own playbook now contradicts shipped SC-FDMA code, and two low-probability DSP nits. None of these were found to cause an observed decode failure in production configuration (loopback/CI stays green throughout, which is itself part of why they survived).

**Bottom line for a release decision:** no blocker. The two medium OTA-wiring items (config-knobs-inert, front-end re-run) are worth a follow-up PR before leaning further on receiver-led rate adaptation in the field; the ACK-wait stall is worth a quick fix since it's operator-visible. Nothing requires blocking work already in flight.

## Full detail by area

### OTA receiver-led rate ladder (highest-value area to fix)

**Config-knobs-inert (#2 above).** Three setters — `set_min_backlog_for_upgrade`, `set_upgrade_hold_frames`, `set_ota_aggressiveness` — all delegate to `self.rate_policy` (`RateAdaptationPolicy`). The daemon reads these from `config.toml` (`ota_min_backlog`, `ota_upgrade_hold_frames`, `ota_aggressiveness`) and logs that they were applied. But the daemon's actual ladder is `self.ota` (`OtaRateController`, `openpulse-core/src/ota_rate.rs`), which decides upgrades purely from SNR floors and NACK count — it has no backlog/hold/aggressiveness concept at all. `RateAdaptationPolicy` is only ever activated by `start_adaptive_session`, which the daemon never calls (only the CLI adaptive demo and `openpulse-testmatrix` do). Compounding this, the backlog value the A2 gate reads (`tx_backlog_bytes`) is written *only* by `set_tx_backlog`, itself called only from the same two non-daemon call sites — so even in the harness the gate only means anything because the harness also feeds it. Net effect: `ota_min_level`/`ota_max_level`/`ota_lock_level` are correctly wired to the real controller; the three "aggressiveness" knobs and the whole backlog/hold gate are decorative on the daemon while claiming otherwise in the startup log.

**Front-end re-run in the OTA decode loop (#3 above).** `ota_decode_and_ack` iterates `rx_candidates` (2 entries whenever `recommended != confirmed`) calling `decode_attempt` per candidate, each of which re-enters `route_audio_stage(InputCapture)` — applying DC-block, notch IIR + persistence-counter `observe()`, DCD, and the stateful streaming AGC — on a burst that was *already* routed once upstream (`accumulate_routed` on the daemon path, or `respond_arq_ota`'s own route). `decode_burst` has an explicit `input_prerouted = true` guard for exactly this reason (added in a prior audit round); `ota_decode_and_ack` never got it. Verified effects, gated behind `agc_enabled`/`notch_enabled` (both default off): wasted repeated DSP passes; the notch's persistence hit-counter advances faster than one-per-burst, which can flip `in_band_interferers()` to "confirmed" and fire the auto-QSY-on-interference path prematurely; transient AGC gain-state pollution across candidates. No evidence this breaks decode itself (demods are amplitude-normalizing and soft LLRs divide out per-block noise variance).

**HARQ retained-LLR pollution across messages (#4 above).** `ota_retained_llrs` combines failed bursts by LLR-vector *length* only; the intended per-session isolation never engages on the daemon because the session id is pinned to the local callsign. This was previously flagged as "MFSK16-only" but the same length-collision condition applies to any soft mode when two consecutive messages happen to share a payload length — notably fixed-size SAR fragments from a file transfer. Consequence: an abandoned message's stale LLRs get MAP-combined into the next same-length message's failing attempts on a bad channel, silently diluting the diversity gain HARQ exists to provide. No false-positive delivery risk (RS/CRC still gates), and the pollution self-limits via the 3-attempt retained buffer, but it undermines the measured 0.43→0.67 HARQ improvement specifically during the bad-channel runs it's meant to help.

### Daemon concurrency around OTA ACK exchange

`ota_send_with_ptt` runs its up-to-4-attempt retry loop synchronously inside the command arm of the daemon's single `tokio::select!` loop; each attempt can block for up to the full `ota_ack_timeout_ms` (9 s for MFSK16 profiles) inside `receive_ota_ack_within`'s `thread::sleep`-polling loop, with no cancellation check. Worst case ~36 s where the control loop cannot process `Abort`/`Disconnect`/mode-change commands (they queue on the bounded 64-slot command channel instead) and cannot service data RX. Separately, `receive_ota_ack_within` opens a *second* capture stream on the same device while the daemon's persistent `rx_stream` sits open and unread for the whole window; on a real (cpal) backend this leaves an unbounded buffer accumulating a large, discontinuous blob that the next RX tick reads as one chunk. Both are real on hardware and both are hidden by `LoopbackBackend`'s synchronous drain semantics, which is exactly why CI doesn't see either.

### DSP / acquisition seam

Two low-severity findings, both structurally real but with limited practical bite: `receive_with_timeout_fec` (CLI-only path) makes its silence-gate and AFC-settle decisions on raw pre-seam audio while the actual decode is seam-routed — self-correcting for the adaptive gate, and the demod already rejects DC, so the realistic harm is wasted attempts, not missed decodes, and it needs a real hardware DC offset or an enabled notch plus in-band tone to matter at all. And a cosmetic disagreement between two hard-decision LLR→bit rules at exactly `+0.0`, essentially unreachable from calibrated float DSP and downstream-gated by CRC/RS either way.

### Test-coverage and traceability gaps

Three items where the missing test is the finding, not a live defect: MFSK16's sub-floor ACK channel has never been tested with `ack_mac_key` set (the FSK4 path is tested; the composition of already-tested pieces on MFSK16 is plausible but unproven); FSK4-ACK's only noisy test asserts the demod is *wrong* at −20 dB rather than *correct* at an operational SNR (indirectly covered at the integration level via `ota_channel_adaptation.rs`, but not at the plugin level); and `scfdma_ce_sweep`'s two tests are permanently `#[ignore]`d with a body that only prints a table and asserts nothing, so the acceptance-table row that cites it is enforced by nobody (a coarser regression is still caught by the non-ignored `scfdma_multipath_timing` test, so this isn't a total gap). Related: `ota_decode_burst` — the function that actually drives daemon receiver-led rate adaptation — is only ever tested with hand-built synthetic `AudioSamples`, never with a burst produced by the real `accumulate_capture` streaming entry point; that missing test is exactly what let finding #3 above ship unnoticed.

### Dead code / dead config surface

`notch_active_freqs`, `clear_tx_session_log`, `enable_afc`, and the unused getters `center_frequency()`/`max_power_watts()` have zero call sites anywhere in the workspace. `SessionDiagnostics.afc_offset_hz` is initialized to `None` and never written or read — every real consumer (CLI, daemon, TUI) calls `engine.last_afc_offset_hz()` directly instead. No functional impact; worth a cleanup pass or explicit wiring if the observability was intended for a UI.

### Docs drift

`CLAUDE.md`'s "Known sharp edges" section states twice, in the load-bearing DSP playbook, that SC-FDMA's `smooth_ce` channel-estimate smoother "was deleted" in PR #689 with "bit-identical" numbers — but the function is live, stateful, and called at 6 sites in `plugins/scfdma/src/demodulate.rs`, feeding every SC-FDMA demod path. This is the one item in this audit that risks actively misleading a future contributor working in exactly the highest-churn area the playbook exists to guide (60+ AFC/carrier commits historically). No functional bug; the doc needs correcting.
